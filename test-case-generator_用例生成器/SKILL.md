# 测试用例生成器

本技能用于从需求文档生成标准化的Markdown格式测试用例，支持转换为Excel和思维导图可视化。

## 使用场景

- 用户提供需求文档（Markdown或文本）并要求生成测试用例
- 用户需要为功能需求创建结构化测试用例
- 用户希望生成可转换为Excel格式的测试用例
- 用户需要兼容markmap思维导图可视化的测试用例

## 文件目录结构

项目使用以下目录结构：

```
markdownToCase/
├── SRS/                    # 需求文档（包含功能点分析）
│   ├── 1. 唯一码收货.md
│   └── ...
├── config/                 # 配置
│   ├── module_config.json  # 模块配置文件
│   └── ...
├── case_MD/                 # 测试用例MD文件（仅测试用例，不含功能点分析）
│   ├── 唯一码收货_测试用例.md
│   └── ...
└── case_excel/              # 生成的Excel测试用例文件
    ├── 唯一码收货_测试用例.xlsx
    └── ...
```

## 输出格式

生成的测试用例遵循以下Markdown结构：

```markdown
> 格式说明放在顶部

# 模块名称（根节点）

## 模块-功能（功能模块）

### 测试用例名称

- [Px] 前置条件 | 操作步骤 | 预期结果
```

## 测试用例格式规则

1. **模块层级**：
   - `#` = 根模块名称
   - `##` = 功能模块（如"扫码处理-解析"）
   - `###` = 测试用例名称（同时作为用例标题）

2. **用例行格式**：

   ```
   - [Px] 前置条件 | 操作步骤 | 预期结果
   ```

   - `[Px]` = 优先级（P0/P1/P2/P3）
   - `|` = 字段分隔符
   - `;` = 多个条件/步骤的分隔符

3. **优先级定义**：
   - **P0（高）**：核心功能、主流程、关键业务场景
   - **P1（中）**：异常流程、错误处理、重要边界场景
   - **P2（低）**：边界测试、极端情况、次要场景
   - **P3（最低）**：共享/复用型测试用例

## 工作流程

### 步骤1：检查历史测试用例（避免重复编写）

**目的**：防止在编写新测试用例时重复编写已有的通用场景

**操作**：

1. 检查 `case_MD/_通用测试用例.md` 是否存在
2. 如果不存在, 自动调用`script/create_folder.py`创建`case_MD/_通用测试用例.md`
3. 如果存在，读取已有的通用场景（从`case_MD/_通用测试用例.md`）
4. 识别当前模块与已有模块的相似场景（如：扫码解析、唯一码验证、提交操作等）
5. 记录可复用的通用场景编号和变量配置

**输出**：生成场景复用清单，包含：

- 可复用的通用场景列表
- 需要替换的变量配置
- 模块特有场景列表

### 步骤2：读取配置

**目的**: 读取模块配置文件，获取模块的通用场景和特有场景

**操作**:

1. 检查 `config/module_config.json` 是否存在
2. 如果不存在, 自动调用`script/create_folder.py`创建`config/module_config.json`
3. 如果存在，读取已有的模块配置（从`config/module_config.json`）
4. 对比步骤1的复用清单，确认需要新增的特有场景

格式参考: `templates/module_config.json`

### 步骤3：读取需求文档

从SRS/目录读取需求文档。

### 步骤4：分析需求

识别功能点并进行详细分析，包括：

- 功能描述
- 输入/输出规范
- 异常情况
- 边界条件
- 业务规则

### 步骤5：插入功能点分析

**在源文档（SRS/）中插入功能点分析段落**，作为需求文档的补充：

输出格式参考: `templates/srs_analysis_template.md`

**功能点分析的作用**：

- 为后续测试用例设计提供清晰的功能点拆分
- 便于测试人员理解需求细节
- 作为测试用例设计的参考依据

### 步骤6：设计用例（基于复用和补充）

**对于唯一码相关模块**：

1. **复用通用场景**：从 `case_MD/_通用测试用例.md` 复制步骤1识别的可复用场景
2. **替换变量**：根据 `config/module_config.json` 中的配置替换变量
3. **补充特有场景**：添加模块特有场景（从配置的 `unique_scenarios` 中获取）

**对于其他模块**：
创建覆盖以下场景的测试用例：

- 正常场景（主流程）
- 异常场景
- 边界条件
- 异常处理

### 步骤7：结构化输出

按模块和功能点组织测试用例，确保每个功能点都有对应的测试场景覆盖。

### 步骤8：分配优先级

用P0/P1/P2/P3标记每个用例，优先级定义：

- **P0（高）**：核心功能、主流程、关键业务场景
- **P1（中）**：异常流程、错误处理、重要边界场景
- **P2（低）**：边界测试、极端情况、次要场景
- **P3（最低）**：共享/复用型测试用例

### 步骤9：保存测试用例

保存到`case_MD/<模块名称>_测试用例.md`.

## Excel转换

**仅在用户明确要求Excel转换时才执行。**

生成的Markdown可以使用配套的Python脚本`scripts/md_to_excel.py`转换为Excel.

脚本将：

- 从 `case_MD/` 目录读取测试用例文件
- 生成Excel文件到 `case_excel/` 目录

## 思维导图可视化

Markdown格式兼容markmap，可以将测试用例结构可视化为思维导图。
